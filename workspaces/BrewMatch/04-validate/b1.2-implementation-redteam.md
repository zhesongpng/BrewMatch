## Findings

- **L1 (LOW, hygiene) тАФ migration test was not regression-marked. FIXED.**
  `test_init_db_adds_bag_columns_to_legacy_table` guards the live-DB backward-
  compat migration but lacked `@pytest.mark.regression`, leaving it deletable by
  a future cleanup. Added the marker; it now runs in the regression suite
  (`pytest -m regression` тЖТ 5 passed).

- **R1 (RISK, documented тАФ NOT a code defect) тАФ Postgres migration path
  unverified in this session.** `DATABASE_URL` is unset locally and no Postgres
  test instance exists, so the `_is_pg` branch (`ALTER TABLE ... ADD COLUMN IF
NOT EXISTS`) is executed only against the live Supabase DB at deploy time. The
  SQL is standard PostgreSQL and mirrors the production-proven `count_brews`
  translation path, but it is unexecuted here. Journaled as a RISK; recommend a
  30-second Supabase Table-Editor check on first B1 deploy. Low blast radius
  (adds one table + two nullable columns; nothing dropped or altered).

## Safe-by-review notes

- `_sqlite_add_missing_columns` uses `f"PRAGMA table_info({table})"` тАФ `table` is
  an internal literal ("users"/"brew_history") from `init_db`, never user input;
  PRAGMA cannot be parameterized. No injection surface.
- `load_brew_history` / `save_brew` unaffected: both name explicit columns; the
  two new nullable columns are ignored until B1.6 wires them.
