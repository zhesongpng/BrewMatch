## Findings

- **F1 (MED → fixed) — cross-user access gap (IDOR class).** `mark_bag_finished`
  and `grams_used_for_bag` keyed on `bag_id` ALONE, unlike every sibling helper
  (`save_brew`, `load_brew_history`, `delete_user_data`) which scope by
  `user_id`. A logged-in user could finish or read the gram-count of ANY bag by
  id, regardless of ownership. Practical risk LOW (bag ids are random 48-bit hex,
  surfaced only to the owner; single-user-per-account personal app) but it
  violates the codebase's scoping convention and `security.md` (access controls,
  no cross-user operations). **Fixed:** both helpers now take `user_id` and add
  `AND user_id = ?` to their WHERE clause; seams pass the session user_id. Two
  negative tests added: `test_mark_finished_scoped_to_user` (u2 cannot finish
  u1's bag) and `test_grams_used_scoped_to_user` (u2's brews don't count toward
  u1's total). Fixed now (not deferred) because B1.6 will reuse
  `grams_used_for_bag` — the scoped signature lands before that consumer.

## Notes (reviewed, not findings)

- **Per-bag connection in `_grams_used`.** The running-low display opens one
  `get_db()` connection per bag per render. At current scale (a handful of bags
  per user) this is acceptable and matches the codebase pattern. In B1.4 the
  query always returns 0 (no brews record a bag yet — that is B1.6). Forward
  note: when B1.6 wires real doses, consider batching grams for all active bags
  into one query if a user accumulates many bags. Not a current defect.
- **Safe no-op when user_id absent.** All seams degrade to `[]` / no-op rather
  than raising if `user_id` is ever missing (covered by `test_no_user_is_safe_noop`).
