# GAP — B1.1 backward-compat test exercised the wrong deserializer

Date: 2026-06-11
Phase: /redteam (B1.1 implementation)

## The gap

B1.1's load-bearing safety claim is "old brew records still deserialize after
roaster/name were added to BeanProfile." That backward-compat behavior lives in
the **persistence** path: `db.py:_deserialize_bean`, which calls
`BeanProfile(**d)` on the stored `bean_json`. The original B1.1 test exercised
`utils.dict_to_bean_profile` instead — a different function that reads each field
via explicit `.get()`. So the function the claim is actually about had zero test
coverage; the test gave false confidence on a sibling path.

## Why it matters

The two functions fail differently. `dict_to_bean_profile` is safe by
construction (named `.get()` per field). `_deserialize_bean` uses `**d` unpacking
— it works for old records (missing keys fall to defaults) and new records
(extra keys are now valid fields), but ONLY because roaster/name were added as
optional fields. A future refactor making either field required would break old
history, and the old test would not have caught it because it never touched the
persistence path.

## Resolution

Added two tests to `tests/unit/test_db.py` on the real db path:

- `test_legacy_bean_json_without_roaster_name_deserializes` (regression-marked) —
  proves a pre-roaster/name `bean_json` deserializes with both fields None.
- `test_brew_roundtrip_preserves_roaster_and_name` — proves a bag-sourced bean
  survives full save→load.

663 unit tests pass; `-W error` clean.

## Lesson

When a backward-compat claim names "deserialize," verify WHICH deserializer the
production path uses. BrewMatch has two (session-state via `utils`, persistence
via `db`) and they are not interchangeable. Test the one the data actually flows
through.
