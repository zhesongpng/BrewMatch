# RISK — B1.2 Postgres migration path is unexecuted until first deploy

Date: 2026-06-11
Phase: /redteam (B1.2 implementation)

## The risk

B1.2 adds the `coffee_bags` table and two `brew_history` columns (`bag_id`,
`actual_dose_g`). The already-deployed Supabase table needs those columns ADDED,
so `init_db` has an `_is_pg(conn)` branch running
`ALTER TABLE brew_history ADD COLUMN IF NOT EXISTS ...`. That branch is **never
executed in this session** — `DATABASE_URL` is unset locally and there is no
Postgres test instance. Only the SQLite migration path is execute-verified.

## Why it is low (but non-zero)

- The SQL is standard PostgreSQL (9.6+) and Supabase is modern Postgres.
- It flows through the same `?`→`%s` translation wrapper (`_PgConn`) that the
  production-proven `count_brews` / `save_brew` paths use.
- `init_db` runs at every app startup (`app.py:348`) against the live connection,
  wrapped in try/except with WARNING-level logging, so a failure is visible in
  the hosting logs rather than silent.
- Blast radius is minimal: one new table + two nullable columns. Nothing is
  dropped or altered; existing data is untouched.

## Mitigation

On the FIRST deploy that includes B1 code, do a 30-second check in the Supabase
dashboard → Table Editor: confirm `coffee_bags` exists and `brew_history` shows
`bag_id` + `actual_dose_g`. If present, the live migration succeeded. If the
startup log shows `database init FAILED`, that is the signal to inspect.

## Why this is a RISK, not a defect

The code is correct by PostgreSQL spec and wired into production startup. The gap
is purely "unexecuted in test" — a verification gap, not a known bug. Recorded so
the deploy-time check is not forgotten. Tracked against B1 deployment.
