# GAP — B1.3 surfaced a pre-existing "Other origin" form bug (fixed)

Date: 2026-06-11
Phase: /redteam (B1.3 implementation)

## The bug

The add-bag form rendered the "Specify origin" text field conditionally INSIDE
`st.form` — `if origin_selection == "Other": st.text_input(...)`. Inside a
Streamlit form, widget values are not committed to session state until the form
is submitted, so the `origin_selection` read during render is always the value
from the PREVIOUS submit. Picking "Other" therefore did not reveal the specify
field until after a submit — and that first submit failed validation with an
empty origin. A user with a non-listed origin needed two submits and saw a
confusing error first.

## Provenance

Pre-existing, not introduced by B1.3. Verified by `git show HEAD:...bean_input.py`
at SHA `6cd9611` (a pre-session commit): the identical conditional-inside-form
pattern was already there. The B1.3 rewrite carried it forward unchanged.

## Fix

Render the specify-origin field unconditionally (labelled "If 'Other', specify
origin"); the submit handler still reads it only when "Other" is selected. Now
works in a single pass. Also escaped the post-save `st.success` toast for
consistency with the bag-list display (output-encoding of user input).

## Lesson

Streamlit `st.form` defers all widget reads to submit — any conditional UI that
depends on another in-form widget's value is broken by construction. Conditional
reveal requires the controlling widget to live OUTSIDE the form (so it reruns on
change), or the dependent field must render unconditionally. Worth checking the
other in-form conditionals in the app (none others found in bean_input).
