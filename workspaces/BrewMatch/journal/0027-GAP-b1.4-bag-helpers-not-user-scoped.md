# GAP — B1.4 surfaced two bag helpers not scoped to the user (fixed)

Date: 2026-06-11
Phase: /redteam (B1.4 implementation)

## The gap

`mark_bag_finished(conn, bag_id)` and `grams_used_for_bag(conn, bag_id)` keyed
on `bag_id` alone. Every sibling persistence helper — `save_brew(conn, user_id,
...)`, `load_brew_history(conn, user_id)`, `delete_user_data(conn, user_id)` —
scopes by `user_id`. These two broke that convention, so a logged-in user could
finish or read the running total of ANY bag by id, regardless of ownership
(an IDOR-class authorization gap).

## Severity

LOW in practice: bag ids are random 12-char hex (48-bit), surfaced only to the
owner via their own bag list, and this is a single-user-per-account personal
app. But it is a real cross-user access gap and violates both the codebase
scoping convention and `rules/security.md` (access controls, no cross-user ops).

## Fix

Both helpers now take `user_id` and add `AND user_id = ?` to their WHERE clause.
The page seams (`_finish_bag`, `_grams_used`) pass the session user_id. Two
negative tests added in `test_db.py`:

- `test_mark_finished_scoped_to_user`: u2 marking u1's bag finished has no effect.
- `test_grams_used_scoped_to_user`: a u2 brew against the same bag_id is not
  counted toward u1's total.

Fixed in-round rather than deferred because B1.6 will reuse `grams_used_for_bag`;
landing the scoped signature now means its next consumer inherits it.

## Lesson

When a helper operates on a per-user resource by its id, scope the query by
user_id even when the id is unguessable. The convention exists for a reason —
new helpers (here, the B1.2 bag CRUD) should follow the sibling helpers'
signatures (`conn, user_id, ...`) by default, and a /redteam grep comparing a
new helper's params against its siblings catches the drift cheaply.
