# GAP — B1.6 no-bag guard was untested until B1.7 red-team

## What

B1.6 added a guard in `brew_session._submit_feedback`: a one-off brew (no bean
picked) must NOT inherit a stale `current_bag_id` from an earlier pick. The guard
was correct but lived inline in Streamlit-bound glue, so no test exercised it.

## Why it matters

A regression there is silent and wrong in a user-visible way: a no-bag brew would
link to a previously-selected bag and decrement that bag's "remaining" count for
coffee the user never pulled from it. Nothing would have failed; the count would
just drift.

## How it surfaced + fix

Found while red-teaming the B1.7 regression test via mutation testing — the loop
test caught storage/scaling breakage but nothing covered the bag-id decision.
Extracted the decision into a pure `_resolve_bag_id(current_bean, current_bag_id)`
helper and added `TestResolveBagId` (3 cases). Mutation (drop the guard → always
return bag_id) now fails the test, confirming it guards the regression.

## Lesson

Correctness-critical branches buried in `st.*` glue are invisible to the test
suite. Extract the decision into a pure function so it can be tested — same
pattern already used for `_scale_recipe` and the bean_input storage seams.
